Table pagination patterns

The first and most vital thing to understand about a UI table is that it could contain more records than the maximum number of rows it exposes to you — where each row represents only one record. In such cases, the table flows its additional rows automatically across as many virtual pages as it must straddle, given the constraint that its number of rows imposes upon it.

This means, a UI table might hide many more records from you than it shows — until you understand how to navigate easily to its hidden records.

Suppose for example that a given table exposes only 20 rows on screen at one time even though it contains 100 records. Pagination controls under the table help you to:

Discover that those hidden records are available and navigable.

Navigate through any of its hidden records to any of its hidden records.

Whenever pagination controls are visibly present under a UI table, you know immediately that the table contains more records than it has rows. After all, when the rendered rows in a table are perfectly sufficient to represent its every record, there is no need for it to straddle multiple pages.

You can tell from the highlighted page number “1” in these pagination controls that only the first subset of table rows is visible on screen currently.

You can click any page number to replace the table’s currently rendered subset of rows with your preferred subset.
